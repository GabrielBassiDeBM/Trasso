## Anti-Patterns Verdict

**LLM assessment**: Doesn't read as AI slop — copy is domain-specific (pt-BR, AP/SAT/ENEM taxonomy), the print preview component shows the real product, and contrast fixes are documented in code comments, all signs of an actual design process. The real failure mode is **typographic inconsistency accumulated through organic growth**: the same design decision (a -0.01em tracking value, a 20px page-title size) gets independently re-typed by hand across many files instead of living in one token, and `font-display` gets applied to UI labels in some files and withheld in structurally identical ones elsewhere with no discernible rule. None of the absolute bans (gradient text, side-stripe borders, eyebrow labels, hero-metric template) are present.

**Deterministic scan**: `detect.mjs` against `src/app` and `src/components` returned exit code 2, one finding — the same one from the prior run: **layout-property animation** in `src/components/banco/BankBrowser.tsx:637` (`transition: max-height`), imported by `BankQuestionCard.tsx`. Still unfixed since the last critique. No false positives.

**Visual overlays**: Not available — no browser automation tool in this environment. All findings are from source/Tailwind-class reading, not rendered pixels; treat contrast, print legibility, and spacing claims as hypotheses to confirm visually.

## Overall Impression

The single biggest finding this pass surfaces that the prior critique missed: **typography in trasso is a system in name only.** The token plumbing exists (`--font-display` / `--font-sans` / `--font-mono`, a `@theme` block, Tailwind's scale) but both font-role tokens point at the same loaded face, so the split is unused scaffolding, and the actual typographic decisions layered on top — tracking, page-title sizing, which headings get the display treatment — are made ad hoc, per file, by hand. The clearest symptom: the "trasso" wordmark itself, the one element that should never visually drift, renders with two different font recipes and three different tracking values across the four places it appears. The second, higher-stakes gap: the printed PDF — trasso's actual product — reuses the screen UI font wholesale with no consideration of print convention, ink economy, or what a Brazilian teacher expects an exam paper to look like. Both are fixable without touching the brand's color identity or component architecture.

## What's Working

1. **Live, scaled print preview beside the editor** (`SheetEditor.tsx`) — directly answers the anxiety this product's core task creates ("will this look right printed?") with immediate visual proof instead of a guess.
2. **Domain-fluent UI** — pt-BR number formatting, AP/SAT/ENEM-specific subject/topic taxonomy with matching icon and color logic, "Gabarito" terminology. This is real product understanding, not generic templating.
3. **A genuinely good landing-page type scale** — `page.tsx`'s heading hierarchy (`text-5xl/3xl/2xl/xl` with responsive step-ups) follows a clean, consistent ratio with `text-wrap: balance` on every heading, a detail most reviews would miss.

## Priority Issues

**[P1] Printed exam body text reuses the screen UI sans (Plus Jakarta Sans) with no print-specific typeface consideration.**
- **Why it matters**: The PDF is the actual deliverable teachers pay for. A geometric sans hinted for screens, at 11pt in dense two-column exam body copy, doesn't match the serif convention set by official exam papers (ENEM, AP, SAT releases) that this audience grades against daily, and its moderate stroke weight costs more ink across repeated class-wide printing than a leaner text face would.
- **Fix**: Trial a well-hinted print serif (Source Serif 4, Lora) for `QuestionRenderer.tsx` and `SheetDocument.tsx` body/passage text only; keep Jakarta for cover titles and all screen UI. Validate the change with a few real teachers before committing — this is a hypothesis worth testing, not an obvious call.
- **Suggested command**: `/impeccable typeset`

**[P2] `font-display` is applied to UI chrome with no consistent rule, and the two font-role tokens (`--font-display`, `--font-sans`) currently point at the identical loaded face.**
- **Why it matters**: `QuestionEditorShell.tsx:137`, `AiReviewPanel.tsx:53,133`, `PageSettingsPanel.tsx:88`, and `PrintToolbar.tsx:19` use `font-display` on per-item labels and section headers, while structurally identical headers in `OrgSettings.tsx` and `BankBrowser.tsx` don't. Because both tokens currently resolve to Plus Jakarta Sans, the inconsistency is invisible today — but it becomes a real, hard-to-track visual bug the day `--font-display` is ever given its own typeface (which the existing two-token scaffolding suggests was the original intent).
- **Fix**: Define one rule — `font-display` only on the wordmark, landing-page editorial headings, and the sheet/document title (cover + editable title input, since that's the artifact itself). Strip it from `QuestionEditorShell`, `AiReviewPanel`'s inline labels, `PageSettingsPanel`, `PrintToolbar`.
- **Suggested command**: `/impeccable typeset`

**[P2] The same hand-typed tracking value and an arbitrary pixel size are copy-pasted across 10+ files instead of living in a token.**
- **Why it matters**: `style={{ letterSpacing: "-0.01em" }}` appears verbatim in `SheetCard.tsx:218`, `ConfirmDialog.tsx:51`, `GenerateModal.tsx:116`, `ScanModal.tsx:113`, `PrintConfigModal.tsx:70`, `BankBrowser.tsx:503,891`, `OrgSettings.tsx:75,88,132,183`, and `DashboardTopbar.tsx:29`. Separately, `text-[20px]` (an arbitrary value identical to Tailwind's own `text-xl`) is independently typed in three of those same files for page titles. Both are the same design decision made by hand repeatedly rather than expressed once — the type scale isn't actually being treated as a system at the component layer, even though it is at the landing-page layer.
- **Fix**: Add a `tracking` token via `@theme` (or just standardize on Tailwind's built-in `tracking-tight` at `-0.025em`) and replace every `text-[20px]` page title with `text-xl`.
- **Suggested command**: `/impeccable extract`

**[P3] The "trasso" wordmark renders with two different font recipes and three different tracking values across the four places it appears.**
- **Why it matters**: A logotype is the one element that should never visually drift by context. `Navbar.tsx` and `(auth)/layout.tsx` use `font-display` with no tracking; `Sidebar.tsx:54` uses plain `font-bold` with an inline `-0.025em` tracking found nowhere else. A user moving from the auth screen to the in-app sidebar sees the brand mark itself change weight/spacing.
- **Fix**: Extract a single `<Wordmark size="sm" | "md" | "lg" />` component with one hardcoded font/weight/tracking recipe, used everywhere "trasso" appears as a logo.
- **Suggested command**: `/impeccable extract`

**[P3] OpenDyslexic (the dyslexia-accessibility font) loads from an external CDN with no print-time load guarantee, carried over from the prior audit but newly scoped here to its typography-loading risk.**
- **Why it matters**: `font-display: swap` means if the print window (`window.open` in `PrintConfigModal.tsx`) renders before the CDN fetch completes, or the CDN is unreachable on school wifi, the page silently falls back to Arial with zero visible signal — defeating the one feature built specifically for accessibility-dependent users, on the population least likely to notice the substitution before paper and ink are already spent.
- **Fix**: Self-host the OpenDyslexic woff2/woff files (freely licensed) and gate the print trigger on `document.fonts.ready` (or a `FontFaceObserver`) when `a11y=1` is set.
- **Suggested command**: `/impeccable harden`

**[P3, carried over] Layout-thrashing `max-height` animation in the question bank browser, still unfixed.**
- **Why it matters**: `BankBrowser.tsx:637` animates `max-height` directly, forcing layout recalculation every frame; flagged by `detect.mjs` in both this run and the prior critique.
- **Fix**: Animate via `transform`/`opacity`, or restructure with CSS grid-template-rows for the height reveal.
- **Suggested command**: `/impeccable optimize`

## Persona Red Flags

**Teacher-in-a-hurry (project-specific)**: The two-column live-preview layout is exactly right for this persona's actual need (proof, not a guess). Risk: `SheetEditor.tsx` uses CSS `zoom: 0.5` for the preview scale, a non-standard property with historically inconsistent Firefox/Safari support — worth confirming on whatever browser a given school's computer happens to run, since this persona has zero patience for a broken preview.

**Accessibility-dependent user (Sam)**: Directly exposed to the OpenDyslexic CDN/load-timing risk above — the one feature built for this persona is also the one with a silent failure mode, and this user is least likely to catch the substitution before printing.

**Jordan (Confused First-Timer)**: No onboarding or guided first-sheet flow found in any reviewed file (Help and Documentation scored 2/4); lands on a calm but minimal empty dashboard with no walkthrough beyond an empty-state CTA.

## Minor Observations

- `SheetDocument.tsx:95-97` hardcodes `color: "#9ca3af"` inline for the print footer credit instead of a CSS variable — the one spot in the print component that skips the token system entirely.
- `QuestionRenderer.tsx` mixes four point sizes (10pt, 10.5pt, 11pt, 13pt) for what's functionally three semantic levels (body / annotation / header); 10 and 10.5 are redundant.
