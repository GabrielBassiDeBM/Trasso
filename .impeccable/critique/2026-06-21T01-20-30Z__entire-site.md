## Anti-Patterns Verdict

**LLM assessment**: This does not read as AI slop overall. There's a real, deliberate design system (named violet/blue tokens, a documented WCAG-AA contrast fix in `globals.css`, dark mode), and the landing page's `SheetPreviewMock.tsx` shows the actual product output instead of a generic hero graphic or stock photo — a genuinely strong, non-default move. Two minor bans were touched: a borderline-earned numbered 01/02/03 "how it works" sequence on the landing page (it IS a real ordered 3-step flow, so this is the explicitly-allowed exception, not a violation), and a light hero-metric pattern on the dashboard (two icon-tinted stat cards). Neither is a real problem on its own.

The actual failure mode here isn't "looks AI-generated," it's **uneven craft**: the marketing page and dashboard got real design attention; the core task surface (SheetEditor, AI modals, PrintConfigModal, Orgs/Invite) did not, most visibly through a complete loss of localization exactly where pt-BR teachers spend the most time.

**Deterministic scan**: `detect.mjs` ran against `src/app` and `src/components` (exit code 2 — findings present). One finding:
- **Layout-property animation** (warning) — `src/components/banco/BankBrowser.tsx:630`: `transition: "max-height 0.28s ..."`. Animating `max-height` causes layout thrash; `BankQuestionCard.tsx` imports this component, so the cost shows up in the question-bank browsing list. Fix: animate via `transform`/`opacity`, or restructure with CSS grid-template-rows for the height reveal.

No false positives to flag — this is a real, fixable performance issue and lines up with the LLM review's general read that the bank/browse surfaces got less polish than the marketing/dashboard layer.

**Visual overlays**: Not available. No browser automation tool is present in this environment, so the browser-injection/overlay step from Assessment B was skipped entirely (fallback signal: no mutable browser tab to inject into). All findings above — including contrast, focus order, animation feel, and responsive breakpoints — are based on reading source/markup/Tailwind classes, not rendered output. Treat heuristics #1, #3, #8, and the accessibility-persona findings as needing live visual confirmation.

## Overall Impression

trasso has a real design system underneath it, not a templated one — the token work, the dark mode, and especially the print-accurate `SheetPreviewMock` on the landing page show actual design intent. But the craft is badly distributed: it's front-loaded onto the marketing page and dashboard, and drops off a cliff exactly where it matters most — the sheet editor and AI modals, the surfaces a teacher spends the most time in, are hardcoded English in a product whose own infrastructure already supports full pt-BR localization elsewhere. The single biggest opportunity is closing that gap: finish the i18n rollout into the editor/AI/orgs surfaces, replace the native `confirm()` at the one truly destructive action in the app, and the score moves from "acceptable" to "good" without touching the brand identity at all.

## What's Working

1. **`SheetPreviewMock.tsx` on the landing page** — a hand-built facsimile of the real print output (lettered MCQ, answer key peeking out behind it) instead of a generic screenshot or abstract icon hero. This is "show the product," not "describe the product," and it's the strongest single design decision in the codebase.
2. **The token system in `globals.css`** — a coherent violet/blue brand pair, full dark-mode parity, and a code comment documenting a real WCAG-AA contrast fix. This is evidence of an actual audit, not a default.
3. **The i18n architecture (`translations.ts`)** — extensive, organized by feature, with pluralization and interpolation already built. The infrastructure for a fully bilingual product exists; it's simply not finished.

## Priority Issues

**[P0] Core workflow and AI features are not localized while the rest of the app is.**
- **Why it matters**: trasso's audience is Brazilian teachers (ENEM/vestibular named explicitly). `SheetEditor.tsx`, `ScanModal.tsx`, `GenerateModal.tsx`, `AiReviewPanel.tsx`, and `PrintConfigModal.tsx` have zero `t()` calls — every string is hardcoded English — while `Sidebar.tsx`, `dashboard/page.tsx`, and `SettingsClient.tsx` correctly use the same `translations.ts` system. A pt-BR teacher gets a bilingual app that flips to English exactly when they're doing the most work.
- **Fix**: Add matching keys to `translations.ts` (already has the right shape and plural helpers) and replace hardcoded strings in the five files above with `t()` calls, mirroring the pattern already correct in `Sidebar.tsx`.
- **Suggested command**: `/impeccable harden`

**[P1] Destructive bulk-delete action uses the unstyled native `confirm()` dialog.**
- **Why it matters**: `SheetsGrid.tsx:45` calls `confirm(msg)` directly — at the single highest-risk interaction in the dashboard, the UI drops entirely out of the brand into unstyled browser chrome, inconsistent with every other dialog in the app (`ScanModal`, `GenerateModal`, `PrintConfigModal` all build custom `role="dialog"` overlays).
- **Fix**: Build a shared `ConfirmDialog` matching the existing modal visual language and use it for single and bulk delete.
- **Suggested command**: `/impeccable harden`

**[P1] Org/invite pages are hardcoded English and inconsistent with the rest of the app.**
- **Why it matters**: `orgs/page.tsx`, `orgs/new/page.tsx`, `orgs/[id]/page.tsx` use raw English with no `t()` calls, and `invite/page.tsx` mixes English UI with a single stray Portuguese string ("Ir para o dashboard") — a tell this surface was bolted on outside the established i18n discipline. This is also where invited colleagues form their first impression of the product.
- **Fix**: Same remediation as the P0 — extend `translations.ts` and wire these four files into the existing pattern.
- **Suggested command**: `/impeccable harden`

**[P2] `GenerateModal.tsx` puts 4 decisions (topic, count, difficulty, question types) in one flat, ungrouped screen.**
- **Why it matters**: Topic, count, difficulty, and a 6-option type-toggle row all sit at equal visual weight with no grouping, despite sensible defaults already being set. A teacher in a hurry has to consciously parse and confirm all four every time.
- **Fix**: Separate "what" (topic) from "how" (count/difficulty/types); collapse the latter into an "Advanced options" disclosure, pre-filled and collapsed by default.
- **Suggested command**: `/impeccable layout`

**[P2] No retry-failed-only path after partial AI-accept failure; AI errors are raw passthrough strings.**
- **Why it matters**: `handleAiAccept` in `SheetEditor.tsx` reports "{n} of {total} added, {failed} failed" with no way to retry just the failed subset. Errors from `ScanModal.tsx`/`GenerateModal.tsx` surface `data.error` verbatim, which can leak server/API phrasing to a non-technical audience.
- **Fix**: Track which questions failed and add a "Retry failed" action; normalize API error strings through `translate()` against known error codes.
- **Suggested command**: `/impeccable clarify`

**[P3] Layout-thrashing animation in the question bank browser.**
- **Why it matters**: `BankBrowser.tsx:630` animates `max-height` directly (`detect.mjs` finding), which forces layout recalculation on every frame of the transition. Low severity but a real, easy perf fix in a list-heavy surface.
- **Fix**: Animate via `transform`/`opacity`, or use `grid-template-rows` for the height reveal.
- **Suggested command**: `/impeccable optimize`

## Persona Red Flags

**Teacher-in-a-hurry (project-specific)**: Subject-matter expert, not a designer, building a test between class periods, usually in Portuguese, frequently interrupted.
- Hits English UI mid-task the moment they open the sheet editor or any AI modal, despite having explicitly chosen Portuguese in Settings.
- `GenerateModal.tsx`'s flat 4-decision form costs time they don't have — no one-click "generate with defaults" path, even though defaults are already pre-selected.
- A partial AI-generation failure leaves them manually figuring out which questions are missing, exactly the busywork the product should eliminate.

**Jordan (Confused First-Timer)**:
- No onboarding, tour, or contextual help anywhere reviewed (Help and Documentation scored 1/4).
- The notification bell in `DashboardTopbar.tsx` is a fully built UI for a feature that's always empty — a wasted discovery click on first use.
- Hardcoded English in core flows reads as "broken/untranslated" to a non-English-fluent first-timer, who has the least tolerance for that kind of friction.

**Riley (Deliberate Stress Tester)**:
- Closing an AI modal (even by clicking the overlay) wipes all generated/scanned results with no persistence or warning.
- `invite/page.tsx` redirects silently to `/dashboard` if the token is missing or stale, with no explanation — confusing for anyone arriving from an old or copied-wrong invite link.

## Minor Observations

- `Navbar.tsx` (marketing) and `Sidebar.tsx` (in-product) are two separate components with separate logged-in-state logic — not a bug, but must be kept in sync manually.
- Dashboard `StatCard`s are a light instance of the hero-metric template; fine as-is, worth simplifying on a future polish pass.
- `EXAM_TYPE_LABELS` in `SheetEditor.tsx` is hardcoded English despite `translations.ts` already having equivalent keys (`examType.test`, etc.) — a clean, already-half-done fix.
- `PrintToolbar`'s alt-version labels are hardcoded across all three print route files.
- The OpenDyslexic accessibility font loads from an external CDN (jsdelivr) rather than being self-hosted — a single point of failure on the one feature where reliability matters most.

## Questions to Consider
